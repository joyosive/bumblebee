# Technical Documentation

## XRPL Transactions Used

BumbleBee uses 8 distinct XRPL transaction types across two layers: direct xrpl.js calls and MCP-XRPL tool calls via the Model Context Protocol.


### All XRPL Transactions

| # | Transaction Type | Bee | Method | Purpose | File |
|---|-----------------|-----|--------|---------|------|
| 1 | `EscrowCreate` | Treasury | xrpl.js | Lock XRP for each milestone with PREIMAGE-SHA-256 crypto-condition and 7-day cancel timeout | `services/escrow.ts` |
| 2 | `EscrowFinish` | Treasury | xrpl.js | Release locked XRP to NGO when milestone is verified — requires cryptographic fulfillment proof | `services/escrow.ts` |
| 3 | `EscrowCancel` | Treasury | xrpl.js | Refund XRP to treasury if milestone expires after 7 days without completion | `services/escrow.ts` |
| 4 | `TrustSet` | Treasury | xrpl.js | Create trustline to RLUSD issuer — required before sending/receiving RLUSD stablecoin | `services/xrplClient.ts` |
| 5 | `Payment` | Treasury | xrpl.js | Send RLUSD (issued currency) to NGO alongside XRP escrow release — dual-currency settlement | `services/xrplClient.ts` |
| 6 | `AccountSet` (DID) | All 5 Bees | MCP | Register Decentralized Identifier for each bee on-chain — enables auditable agent identity | `services/mcpClient.ts` |
| 7 | `OracleSet` | Reviewer | MCP | Publish trust score on-chain as Oracle price data — queryable by any XRPL application | `services/mcpClient.ts` |
| 8 | `CredentialCreate` | Reviewer | MCP | Issue campaign completion credential to NGO on-chain — verifiable proof of impact delivery | `services/mcpClient.ts` |

## Transaction Flow by Campaign Phase

### Phase 1: Fund (TreasuryBee)

```
Treasury Wallet ──► EscrowCreate (M1: 33% XRP, condition=SHA256_hash_1)
                ──► EscrowCreate (M2: 33% XRP, condition=SHA256_hash_2)
                ──► EscrowCreate (M3: 34% XRP, condition=SHA256_hash_3)
                ──► EscrowFinish (M1: auto-release seed funding)
                ──► Payment     (M1: RLUSD equivalent to NGO)    [if RLUSD enabled]
```

**On-chain artifacts:** 3 escrow objects created, M1 immediately released, 2 remaining locked.

### Phase 2: Verify & Release (VerifierBee + TreasuryBee)

```
For each milestone (M2, M3):
  NGO submits evidence via Telegram
  VerifierBee analyzes with LLM
  If approved:
    Treasury Wallet ──► EscrowFinish (fulfillment=preimage)
                    ──► Payment     (RLUSD to NGO)               [if RLUSD enabled]
  If rejected:
    No transaction — milestone status set to revision_needed
```

### Phase 3: Score & Certify (ReviewerBee)

```
Reviewer Wallet ──► OracleSet       (trust score as price data)
                ──► CredentialCreate (campaign completion cert)
```

### Phase 4: Expire (Automatic — Hourly Check)

```
If milestone not completed within 7 days:
  Treasury Wallet ──► EscrowCancel (funds return to treasury)
```

## Crypto-Conditions (RFC PREIMAGE-SHA-256)

Each escrow uses a PREIMAGE-SHA-256 crypto-condition pair:

```
Condition   = SHA-256(fulfillment)     ← stored on-chain in EscrowCreate
Fulfillment = random 32-byte preimage  ← stored in database, submitted in EscrowFinish
```

- Generated by `five-bells-condition` library (RFC draft-thomas-crypto-conditions-04)
- Each milestone gets a unique condition/fulfillment pair
- XRPL validates `hash(fulfillment) === condition` before releasing funds
- Fulfillments are never exposed until the milestone is verified

## RLUSD Dual-Currency Flow

When `RLUSD_ISSUER` is configured, BumbleBee operates in dual-currency mode:

```
                    ┌─────────────────────────┐
                    │   Milestone Verified     │
                    └────────┬────────────────┘
                             │
              ┌──────────────┼──────────────┐
              ▼                              ▼
    EscrowFinish (XRP)              Payment (RLUSD)
    Trustless release               Stablecoin settlement
    via crypto-condition            via issued currency
              │                              │
              ▼                              ▼
    NGO receives XRP               NGO receives $USD
    (on-chain proof)               (no volatility risk)
```

**Why dual-currency:**
- XRP escrows provide trustless, conditional fund release (the security layer)
- RLUSD provides stable-value payments NGOs can actually spend (the utility layer)
- Both transactions are on-chain and auditable

**Setup requirements:**
1. Treasury and NGO wallets need TrustLines to the RLUSD issuer (`TrustSet`)
2. Treasury must hold RLUSD balance
3. Trustlines are created automatically on startup via `ensureRLUSDTrustlines()`

## Wallet Architecture

5 isolated wallets — one per bee — each with its own XRPL account and registered DID:

```
┌─────────────────────────────────────────────────────────────┐
│ XRPL Testnet                                                │
│                                                             │
│  Treasury (rXXX...)    ── Holds pool, creates escrows       │
│  Facilitator (rXXX...) ── Registers campaigns via DID       │
│  Evaluator (rXXX...)   ── Records evaluation scores         │
│  Verifier (rXXX...)    ── Issues verification records       │
│  Reviewer (rXXX...)    ── Publishes Oracle + Credentials    │
│                                                             │
│  NGO Wallet (rXXX...)  ── Receives escrow releases + RLUSD  │
└─────────────────────────────────────────────────────────────┘
```

**Why separate wallets:**
- Full auditability — each bee's actions are traceable on-chain
- DID registration — each bee has a verifiable identity
- Least privilege — Treasury holds funds, others cannot move money
- Parallel operations — no transaction sequence conflicts

## MCP Integration

The Model Context Protocol connects each bee to the XRPL via a stdio transport:

```
Bee Process ──► StdioClientTransport ──► mcp-xrpl server process
                                          │
                                          ├── did-create      (AccountSet)
                                          ├── oracle-set      (OracleSet)
                                          ├── credential-create (CredentialCreate)
                                          ├── escrow-create   (EscrowCreate)
                                          ├── escrow-finish   (EscrowFinish)
                                          ├── escrow-cancel   (EscrowCancel)
                                          ├── get-account-info (account_info)
                                          └── ... 40+ more tools
```

Each bee spawns its own MCP server process with its own wallet seed via environment variable injection. If MCP is unavailable, the system falls back to direct xrpl.js calls.

## Database Schema

```sql
campaigns ──┬── milestones     (3 per campaign, sequential activation)
            ├── escrows        (1 per milestone, PREIMAGE-SHA-256 locked)
            ├── fund_allocations (audit trail of pool withdrawals)
            └── trust_scores   (final scores with Oracle + Credential tx hashes)
```

| Table | Key Fields | Purpose |
|-------|-----------|---------|
| `campaigns` | id, ngo_name, ngo_telegram_id, title, sector, country, funding_goal, status, evaluation_score | Campaign lifecycle tracking |
| `milestones` | campaign_id, number, title, status, evidence_file_ids, feedback | Milestone progress with evidence |
| `escrows` | campaign_id, milestone_number, tx_hash, sequence, condition, fulfillment, amount, status | On-chain escrow records |
| `fund_allocations` | campaign_id, amount, treasury_balance_before | Audit trail |
| `trust_scores` | campaign_id, score, breakdown, oracle_tx_hash, credential_tx_hash | On-chain reputation |

| `/api/events` | GET | Last 100 agent events |
| `/health` | GET | Server status, connected clients, event count |
